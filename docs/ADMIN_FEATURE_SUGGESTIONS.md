# 管理端可增加功能建议

**状态说明**：下表 1～7 项均已实现（订单列表、会员导出 CSV、推荐奖励配置、待审充值提醒、操作审计日志、用户详情订单/流水、流水双表一致性）。本文档保留原建议描述供参考。

以下为最初检查时的建议描述（抢单相关已排除）。

---

## 一、订单列表（订单/抢单记录）

**现状**：  
- 仪表盘和报表里用 `orders` 表算「今日收益」「总收益」；  
- 重置进度时会取消用户的 pending 订单；  
- 没有在管理端集中查看「所有订单」的页面。

**可增加**：  
- 新增菜单例如「**订单记录**」或「抢单记录」；  
- 接口：`GET /api/admin/orders`，支持按用户 ID、用户名、状态（pending/completed/cancelled）、日期范围筛选、分页；  
- 列表字段：订单号、用户、金额、佣金、状态、来源（match/start/dispatch）、创建时间等；  
- 用途：对账、排查用户「卡单」、看完成量/取消量。

---

## 二、会员列表导出

**现状**：  
- 已有「导出推荐数据」(`/export-invites`)，导出的是邀请关系/推荐数据；  
- 没有「把当前会员列表导出成 CSV」的功能。

**可增加**：  
- 在会员列表上方增加「**导出当前列表**」按钮；  
- 接口：`GET /api/admin/users/export`，沿用当前列表的筛选条件（search、status、vip_level、日期等），返回 CSV；  
- 字段可包含：ID、用户名、手机、余额、冻结、邀请码、推荐人、团队数、抢单状态、进度、状态、注册时间等（敏感信息如密码、资金密码不导出）；  
- 用途：线下统计、备份、报税等。

---

## 三、推荐奖励金额配置入口

**现状**：  
- 注册时给推荐人发奖励，金额从 `settings.referral_reward_amount` 读，没有则默认 5；  
- `user.js` 里会确保该 key 存在；  
- 管理端「系统」下的设置项较多，需确认是否已有该键的表单项。

**可增加（若当前没有）**：  
- 在「系统」里（例如规则与内容 / 财务与提现）增加一项「**推荐奖励金额**」(referral_reward_amount)，单位与现有逻辑一致；  
- 保存进现有 `settings` 表即可，注册逻辑已在读该 key；  
- 用途：运营可自行改「拉新奖励」金额，无需改代码。

---

## 四、仪表盘待审充值提醒

**现状**：  
- 财务审批有「待审核提现」数量角标、仪表盘有「最新提现申请」；  
- 待审核充值也有接口和数据，但仪表盘没有对称的「待审核充值」提醒或列表。

**可增加**：  
- 在仪表盘或财务入口处增加「待审核充值」数量展示（或与提现一起做成「待审：充值 x / 提现 x」）；  
- 可选：在仪表盘加一块「最新充值申请」列表，与「最新提现申请」类似，点击可跳转到财务-充值审批；  
- 用途：避免只关注提现、忽略充值待审。

---

## 五、操作审计/操作日志

**现状**：  
- `db.js` 里已有 `audit_logs` 表（actor_id, action, entity_type, entity_id, reason, metadata, created_at）；  
- 管理端目前没有「操作日志」菜单，也没有在关键操作（调余额、改状态、审批、编辑用户等）里写入 `audit_logs`。

**可增加**：  
- 在关键操作处写入 `audit_logs`（操作人、动作、对象类型、对象 ID、备注等）；  
- 新增菜单「**操作日志**」，接口 `GET /api/admin/audit-logs`，支持按操作人、动作类型、日期筛选、分页；  
- 用途：追责、合规、排查「谁改过余额/状态」。

---

## 六、用户维度的订单与流水

**现状**：  
- 会员列表可进「资金」「密码」「派单」「编辑」「推荐」「重置」「冻结」等；  
- 没有「该用户最近订单」「该用户最近流水」的集中查看入口；  
- 流水明细页可按 user_id 筛选，但需要自己输入 ID。

**可增加**：  
- 在用户操作列增加「**详情**」或「订单/流水」入口，打开弹窗或抽屉；  
- 弹窗内：该用户基本信息 + 最近 N 条订单（调用 orders 表）+ 最近 N 条流水（调用现有 transactions 或 ledger）；  
- 接口可复用或扩展：如 `GET /api/admin/users/:id/orders`、`GET /api/admin/users/:id/transactions`，或在一个详情接口里一起返回；  
- 用途：不用再切到流水页输 user_id，查单、查账更快。

---

## 七、流水表与数据一致性（可选）

**现状**：  
- 存在 `ledger` 表（部分调余额、提现/充值审批会写）；  
- 存在 `transactions` 表（任务完成、部分审批会写）；  
- 报表「资金流水明细」读的是 `transactions` 表。

**建议（可选）**：  
- 若希望管理端看到的流水更完整，可考虑：  
  - 要么在写 `ledger` 的关键操作时同步写 `transactions`，  
  - 要么在「资金流水明细」里同时查两表并合并展示（需去重与排序）；  
- 不做也不影响现有功能，只是流水「完整性」和「一处查」的体验问题。
